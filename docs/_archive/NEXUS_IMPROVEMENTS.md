# Nexus 平台不足分析与改进建议

> 基于 CCDesk 项目端到端验证的真实使用经验
> 日期：2026-03-19

---

## 🔴 P0 — 核心功能缺失

### 1. 缺少断点续传 / 任务重试机制
- **现象**：Phase 2C 超时后，CC 已写了 1071 行代码但 cargo check 有 1 处编译错误。超时直接标记 failed，无法从断点继续。
- **影响**：大任务（>300s）一旦超时，之前的工作全部浪费，需要重新提交。
- **改进建议**：
  - 任务支持 `resume` 参数：重试时注入上次的输出和状态
  - 超时前 60s 发预警事件，允许延长超时
  - CC 输出的中间文件（代码）自动暂存，失败后可恢复

### 2. 缺少任务输出实时查看能力
- **现象**：CC 执行期间，主 Agent 只能轮询 `GET /api/gateway/tasks/{id}` 查看最终结果。无法看到 CC 的实时输出（正在写哪个文件、cargo check 进度等）。
- **影响**：600s 超时期间完全黑盒，无法判断 CC 是在正常工作还是卡住了。
- **改进建议**：
  - WebSocket 推送任务进度事件（类似 CC 的 `assistant`/`tool_use` 消息）
  - 或提供 `GET /api/gateway/tasks/{id}/output` SSE 端点
  - 前端任务详情页实时展示 CC 输出流

### 3. 缺少任务依赖 / 工作流串联 API
- **现象**：Phase 之间有依赖关系（2B 依赖 2A，2C 依赖 2B），但只能手动逐个提交。Nexus 有 workflow 引擎但缺少 API 层的"串行任务组"概念。
- **影响**：需要外部 Agent 编排多 Phase 执行，Nexus 只承担单任务调度角色。
- **改进建议**：
  - 支持 `depends_on` 参数：任务自动等待依赖完成后再路由
  - 或提供"任务链"API：一次提交多个有序任务

## 🟡 P1 — 体验问题

### 4. 任务超时策略不灵活
- **现象**：固定超时（10-3600s），超时直接 kill。编译型任务（cargo check）本身可能需要 60s+，加上 CC 思考和编码，容易误杀。
- **影响**：Phase 2C 写了代码但没时间运行 cargo check 验证。
- **改进建议**：
  - 支持"软超时"：发 SIGINT 让 CC 优雅退出（提交已有代码），硬超时才 SIGKILL
  - 按 agent_type 设置默认超时（CLI 类型默认 600s）
  - 支持 on_timeout 策略：`kill` / `graceful` / `extend`

### 5. Bridge 负载均衡策略缺失
- **现象**：多 Bridge 时只有简单选择，没有考虑：
  - Bridge 当前负载（正在执行几个任务）
  - Bridge 历史成功率
  - 任务与 Bridge 的亲和性（同一项目的任务路由到同一 Bridge）
- **改进建议**：
  - 加权轮询（基于并发数）
  - 项目亲和路由（同项目任务 → 同 Bridge，利用文件缓存）

### 6. 任务结果不够结构化
- **现象**：CC 输出是原始 stream-json 行拼接的字符串，需要手动解析才能判断结果。
- **影响**：自动化验收困难——需要解析 CC 输出来判断是否编译通过、是否 commit。
- **改进建议**：
  - 解析 CC stream-json，提取关键事件：文件修改列表、命令执行结果、错误信息
  - 返回结构化 result：`{ files_modified: [...], commands_run: [...], exit_code: N, summary: "..." }`

### 7. 前端任务详情页信息不足
- **现象**：任务详情页只显示基本状态、耗时、原始输出。缺少：
  - CC 修改了哪些文件
  - cargo check / npm build 是否通过
  - 是否有 commit
  - 工具调用统计（写了几个文件、执行了几次命令）
- **改进建议**：
  - 解析 CC 输出，提取文件修改列表展示
  - 自动检测编译/构建结果
  - 增加"文件变更"标签页

## 🟢 P2 — 增强功能

### 8. 任务优先级队列
- **现象**：`priority` 参数（high/normal/low）已支持，但实际只是 metadata，没有真正的优先级队列。所有 pending 任务 FIFO 处理。
- **改进建议**：实现优先级队列，high 任务优先路由。

### 9. 任务模板 / prompt 模板
- **现象**：每次提交 CCDesk 任务都需要写完整的 prompt（500+ 字），重复度高。
- **改进建议**：
  - 支持 prompt 模板：`template: "cc-desk-phase"`, `variables: { phase: "2A" }`
  - 项目级模板配置（CLAUDE.md 路径、验收命令等）

### 10. Bridge 健康检查增强
- **现象**：Bridge 在线但 CC CLI 可能已损坏（路径变了、版本降级了）。任务提交后才发现问题。
- **改进建议**：
  - Bridge 启动时验证 agent 配置（CC CLI 可执行、版本匹配）
  - 定期心跳时附带 agent 健康状态
  - 任务失败率超过阈值自动降级 Bridge

### 11. 任务成本追踪
- **现象**：不知道每个任务消耗了多少 LLM tokens / 费用。CC 的 stream-json 输出里包含 `usage` 字段但 Nexus 没有提取。
- **改进建议**：
  - 解析 CC 输出中的 usage 信息
  - 汇总到任务 record：input_tokens, output_tokens, cost_usd
  - 项目/Agent 维度成本统计

### 12. 任务日志持久化增强
- **现象**：任务输出存在数据库里但缺少搜索、过滤能力。大任务输出可能达到 100KB+。
- **改进建议**：
  - 输出分页查看
  - 按时间/类型过滤（只看 tool_use、只看 errors）
  - 支持下载原始 stream-json

## 🎯 建议的下一步

1. **P0-2 实时输出** — 优先级最高。可以在 oc-bridge 层直接把 CC 的 stream-json 事件转发到 Gateway WebSocket，前端实时展示。改动量小，效果显著。
2. **P0-1 断点续传** — 实现任务结果暂存 + resume API。
3. **P1-6 结构化结果** — 解析 CC 输出，提取文件变更和命令结果，为自动化验收打基础。
4. **P1-4 灵活超时** — 支持 graceful shutdown 策略。
